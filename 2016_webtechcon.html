Part I: Introduction to SPAs

Classic Web Application

  • Browser sends HTTP-Request and receivers HTML
  • Content is rendered on the server side
  • Each interaction repeats this cycle
  • UI/UX restricted by this rendering cycle
  • Pure form hardly seen anymore
  • Minor UI-changes can be done on client side using JavaScript snippets
  • Sometimes full blown Single-Page Applications sort of happen as parts of Classic Web Applications

Classic Web Application

Single-Page Applications (SPAs)

  • only a single page containing the application is loaded into the browser
  • all rendering and controll now in the browser using JavaScript
  • thus, major parts of the application now in the browser

SPA #1

SPAs to an extreme

  • you can also move (parts of) the business logic into the browser
  • allowing for offline option
  • you can even go as far as to move the complete application into the browser

SPA #2

Part II: Introducing SPA frameworks

Why not just jQuery?

  • jQuery is very good at changing sets of existing elements
  • good if there already are existing elements
  • unobtrusive
  • not so good, if there is nothing to begin with
  • not so good for controlling page flow

Why use a standard Framework?

  • avoid the risk of accidentally creating your own
  • use something proven and tested
  • community, documentation
  • easily find skill and talent

Your Options

  1. The Classic: Angular 1.x
  2. From Anti-Patterns to Best-Practices: React
  3. One for all: Angular 2
  4. The Standard: Web Components
  5. (I know there is more)

Angular 1

  • The classic from Google
  • Plenty of developers, discussion groups, integrations, etc.
  • 3 Ds
    • Data Binding (2-Way)
    • Dependency Injection
    • Directives (=== Components)

React

  • Framework for Facebook and many more
  • Components encapsulate template and logic
  • Abstraction from DOM: f(model) -> UI
  • Minimal API
  • One-Way-Data-Binding: When state changes, components re-render
  • Uses virtual DOM to gurantee decent performance
  • Can render on client and on server

Angular 2

  • Successor of Angular 1
  • Announcement (fall 2014) has led to irritations
    • Concepts very different from Angular 1
    • No migration path (at that time)
  • Developed in Microsoft's TypeScript (ES5, ES6 and Dart also supported)
  • Optimised for speed
  • Server side rendering also supported

Web Components

  • Components as a Standard
    1. Custom Elements: define your own tags
    2. HTML Templates: simple templates as part of a HTML page (now in HTML Standard)
    3. Shadow DOM: Local encapsulation of DOM and CSS (will be moved into DOM/HTML Standard
    4. HTML Imports: Importing of HTML fragments / Web Components

Web Components: procedural, free style (here ES6)


class Greeter extends HTMLElement {
  constructor() {
    this.attachShadow({mode: 'closed'}).innerHTML = template;
    this.shadowRoot.querySelector("#in").addEventListener("keyup",
        event => this.setModel(event.target.value));
    this.setModel(this.getAttribute("greeting"));
  }
  attributeChangedCallback(name, oldValue, newValue) {
    if (name === "greeting") {
      this.setModel(newValue);
    }
  }
  setModel(value) {
    this.shadowRoot.querySelector("#log").textContent = value;
    this.shadowRoot.querySelector("#in").value = value;
  }
}


document.define('greeter-element', Greeter);


<greeter-element greeting="Hello"></greeter-element>

@Component({
  selector: 'greeter'

        this.greeting = '';
        input.focus();
    }
}

}

Criterion: Architecture and Model

Angular 1

  • 2-Way Data-Binding with Mutable State!
  • Convoluted change detection
  • Automatic updates of UI
  • Good for simple forms

2-Way Data-Binding with Mutable State

A good idea?

React

  • 1-Way Data-Binding
  • Immutable state recommended (Immutable.js)
  • Data should flow in hierarchy only
  • Model is special state-property of component
  • Re-Rendering triggered by setState
  • Clear and fast change detection on level of virtual DOM
  • Flux Architecture for larger apps

Angular 2

  • 1-Way Data-Binding
  • Immutable state recommended (Immutable.js)
  • Data flows inside of clear hierarchy (@Input, @Output)
  • Model can be any property of component
  • Clear and fast change detection on model level
  • Re-Rendering triggered when model-change is detected, using zones
  • https://github.com/ngrx/store for larger apps
  • Observables for events

Web Components

  • Whatever you want as a model
  • Might be tricky to synchronize Properties and Attributes
  • no change detection of model or automatic re-rendering

Wrap-Up

  • use SPAs for a highly interactive (part of) app or need of offline
  • use Angular 1.x for simple form based app or not much JavaScript knowledge
  • use Angular 2 for everything and nothing or if you want to use TypeScript
  • use React for a real app and if you like to choose for own stack
  • use Web Components if you like procedural code and trust in the standard
